# Чат-колонка виджета (DeepSeek через автопилот)

Дата: 2026-07-08

## Цель

Добавить в виджет вторую, скрываемую колонку — чат с LLM (DeepSeek). Весь текущий
интерфейс виджета становится левой колонкой без изменений; справа появляется чат-панель,
которую можно открыть/скрыть. Скрыт чат → окно снова одна колонка, как сейчас.

Чат — **общий ассистент** (спросить что угодно), со своим системным промптом, без
рабочего контекста профиля/вакансий. LLM переиспользуется из автопилота
(`~/projects/work-autopilot`, `src/autopilot/llm.py`) — виджет не ходит в API напрямую,
а шеллит бинарь автопилота, как это уже сделано в кнопке «Ответить HR» (`src/hr_reply.rs`).

## Решения (зафиксированы на брейнсторминге)

- **Назначение:** общий ассистент, свой системный промпт, без спец-контекста.
- **Картинки:** DeepSeek (`deepseek-chat`) — текстовая модель, изображения не принимает.
  Обходим локальным OCR (tesseract): из картинки достаём текст и шлём его как текст.
- **История:** эфемерная, в памяти виджета. При перезапуске — чистый лист. Есть кнопка «очистить».
- **Бэкенд:** без долгоживущего процесса. На каждое сообщение шеллим `autopilot chat`
  (холодный старт автопилота ≈ 0.30s — пренебрежимо на фоне ответа LLM в секунды).
- **Вёрстка:** правая `SidePanel::right` egui; текущий контент остаётся в `CentralPanel`
  и автоматически становится левой колонкой.

## Предпосылки (вне кода, поставить отдельно)

- **`tesseract-ocr-rus`**: сейчас в системе есть только языки `eng` и `osd`. Для OCR
  русских скриншотов нужен русский пакет. Без него OCR русского текста даст мусор.
  OCR запускаем с языками `rus+eng`.

## Архитектура

### 1. Бэкенд — новая подкоманда `autopilot chat` (проект work-autopilot)

- Добавить `chat` в `choices` парсера `src/autopilot/__main__.py`.
- Обработчик: читает из stdin JSON `{"messages": [{"role": "user"|"assistant", "content": "..."}, ...]}`
  — всю историю диалога; печатает ответ ассистента в stdout. Логи — в stderr, stdout чистый
  (та же дисциплина, что у `reply-hr`).
- В `src/autopilot/llm.py` (`QwenBrain`) добавить метод `chat(messages: list[dict]) -> str`,
  принимающий готовый список сообщений (сейчас есть только `complete(system, user)`).
  Системный промпт «общий ассистент» подставляется первым сообщением на стороне обработчика
  команды (или внутри метода — реализационная деталь). Ретраи/таймауты — как в существующем
  `complete` (tenacity, 3 попытки).
- Никаких изменений в существующих командах и в клиенте, кроме нового метода.

### 2. Виджет — новый модуль `src/chat.rs`

По образцу `src/hr_reply.rs`.

Ответственность:
- Хранит историю: `Vec<ChatMessage { role: Role, text: String }>` (Role = User | Assistant | Error).
- Хранит состояние ввода: текущий текст поля и опциональный «прикреплённый OCR-текст».
- `send(...)`: в фоновом потоке сериализует историю (user/assistant, без Error-строк) в JSON,
  шеллит `autopilot chat` (cwd = autopilot_dir, bin = autopilot_bin — из `Config`), пишет JSON
  в stdin, читает ответ из stdout. По завершении добавляет ответ (или Error) в историю и будит
  перерисовку `ctx.request_repaint()`. UI не блокируется.
- Состояние запроса: `Idle | Sending` — пока `Sending`, в ленте строка «…думает», поле ввода
  и кнопка отправки заблокированы (или отправка игнорируется).
- `clear()`: очистить историю и ввод.

Интерфейс для UI: методы «добавить сообщение пользователя», «отправить», «очистить»,
«снимок истории для отрисовки», «идёт ли запрос».

### 3. OCR картинок

- Кнопка 📎: `wl-paste -t image/png` → временный файл в scratch/`/tmp` → `tesseract <img> stdout -l rus+eng`
  → текст. Текст кладётся как «прикреплённый OCR-текст» (чип «📷 текст с картинки» рядом с полем ввода).
- Drag-drop: egui `ctx.input(|i| i.raw.dropped_files)` — если бросили файл-картинку в область чата,
  тот же путь OCR.
- При отправке: если есть прикреплённый OCR-текст, он добавляется к тексту сообщения
  (напр. `"<текст пользователя>\n\n[текст с картинки]:\n<ocr>"`), затем чип сбрасывается.
- Ошибки OCR (нет tesseract / пустой результат / нет картинки в буфере) → короткая красная
  строка-статус у поля ввода, сообщение не отправляется.

### 4. Вёрстка (src/main.rs)

- Новое поле состояния `App`: `chat: chat::ChatState`, `chat_open: bool`, и запомненная
  «одноколоночная» ширина окна `width_one_col: Option<f32>`.
- Кнопка-переключатель 💬 в строке заголовка рядом с 📌: тогглит `chat_open`.
- При смене `chat_open`:
  - открытие: запомнить текущую ширину окна, запросить `ViewportCommand::InnerSize` = ширина+`CHAT_W`;
  - закрытие: запросить `InnerSize` обратно на запомненную одноколоночную ширину.
  - `CHAT_W ≈ 340.0` (константа).
- Рендер: если `chat_open`, до `CentralPanel` показать `egui::SidePanel::right("chat")`
  (resizable, дефолтная ширина `CHAT_W`) с содержимым чата. `CentralPanel` с текущим
  контентом рисуется как сейчас — он и есть левая колонка.
- Содержимое чат-панели (сверху вниз):
  - шапка: «💬 Чат» + кнопка «очистить»;
  - `ScrollArea` с лентой сообщений (`stick_to_bottom`), роли различаются цветом/выравниванием,
    Error — красным; в конце строка «…думает» при активном запросе;
  - нижняя строка: multiline-поле ввода + 📎 (вложить) + ▶ (отправить). **Enter** — отправить,
    **Shift+Enter** — перенос строки; чип «📷 …» показывается над строкой ввода, если есть OCR-вложение.

## Поток данных

```
ввод пользователя
   │  (+ опц. картинка → wl-paste/drag → tesseract rus+eng → OCR-текст)
   ▼
история [user, assistant, user, …] + новое user-сообщение
   ▼  JSON в stdin
autopilot chat  ──►  llm.py QwenBrain.chat(messages)  ──►  DeepSeek API
   ▼  ответ в stdout
добавить assistant-сообщение в ленту  ──►  request_repaint
```

## Обработка ошибок

- Нет бинаря/сети/пустой ответ LLM / ненулевой код autopilot → Error-сообщение в ленте
  (последняя строка stderr автопилота как деталь), история не рвётся.
- OCR: нет `tesseract` / нет картинки в буфере / пустой OCR → красный статус у поля ввода,
  отправка не происходит.
- Отсутствие `rus`-пакета не роняет OCR (tesseract отработает по `eng`), но качество на русском
  будет плохим — это предпосылка развёртывания, не рантайм-ошибка.

## Тестирование

- **work-autopilot:** unit-тест обработчика `chat` — на вход JSON-история, LLM-клиент замокан,
  проверяем формирование `messages` (системный промпт + история) и что stdout = ответ, stderr
  отделён. Тест метода `QwenBrain.chat` на формирование запроса (клиент замокан).
- **Виджет (`chat.rs`):** unit-тесты чистой логики без GUI и без сети:
  - сериализация истории в JSON для stdin (Error-строки исключаются);
  - склейка сообщения с OCR-вложением;
  - `clear()` сбрасывает историю и ввод;
  - состояние `Sending` блокирует повторную отправку.
- Внешние процессы (`autopilot`, `wl-paste`, `tesseract`) в юнит-тестах не запускаем — логику
  запуска изолируем за тонкой границей, как в `hr_reply.rs`.
- Ручная проверка: открыть чат (окно расширяется), отправить сообщение, получить ответ,
  вставить скриншот с текстом, скрыть чат (окно сужается до одной колонки).
