# Веб-микрофон: текст и картинки со страницы → лента в окне hw-web

## Зачем

Со второго устройства во время разговора хочется не только говорить, но и
перекидывать на основной комп текст (ссылки, куски кода) и картинки
(скриншоты). Виджет показывает их в том же плавающем окне `hw-web`,
рядом с транскрипцией, только для просмотра.

## Как пользоваться

1. На странице веб-микрофона под лентой распознавания — поле ввода:
   Enter или «➤» отправляет текст; «📎» открывает выбор файла;
   Ctrl+V вставляет картинку прямо из буфера (скриншот).
2. В окне `hw-web` две секции: сверху «🗣 Речь» (транскрипция, как раньше),
   снизу «📥 Присланное» — тексты и картинки хронологической лентой,
   каждая секция со своим скроллом.
3. Окно `hw-web` ресайзится за грип в правом нижнем углу; размер
   запоминается между запусками.

## Архитектура

- **Эндпоинты** (`src/webmic.rs`, токен-правило как у `/api/audio` —
  `needs_token` уже покрывает весь `/api/*`):
  - `POST /api/msg` — тело: сырой UTF-8 текст, лимит 64 КБ.
    Пустой/непарсящийся текст игнорируется (`400`).
  - `POST /api/img` — тело: сырые байты картинки, `Content-Type:
    image/png|jpeg|webp`, лимит 8 МБ. Декод крейтом `image`
    (уже в дереве через eframe, добавляется прямой зависимостью)
    сразу при приёме; даунскейл до вписывания в 1600×1600
    (filter Triangle). Не decoded → `415`; больше лимита → `413`.
    Ответ обоих — `{"ok":true}`.
- **Модель** — в `Shared` добавляется `posts: VecDeque<Post>` и счётчик
  `next_post_id: u64`:
  `enum Post { Text(u64, String), Image(u64, egui::ColorImage) }`
  (id нужен виджету как ключ кэша текстур). Кап — 30 постов,
  старые выкидываются спереди.
- **Окно hw-web** (`src/main.rs`) — `with_resizable(true)`,
  `with_min_inner_size([320, 280])`, размер в `state.rs`
  (`web_w`/`web_h`, дефолт 420×320) по образцу чата
  (`chat_spawn_size`/`chat_size`), грип `draw_resize_grip`.
  Верхняя секция — транскрипция (как сейчас), нижняя половина —
  «📥 Присланное»: `ScrollArea` со `stick_to_bottom`, тексты лейблами,
  картинки текстурами (`TextureHandle` создаётся при первом показе,
  кэш `HashMap<u64, TextureHandle>`, вписывание по ширине окна
  с сохранением пропорций). Только просмотр — без кликов и действий.
  Кэш текстур чистится от id, которых больше нет в ленте.
- **Страница** (`web/src/App.jsx`) — под лентой: `<input>` текста
  (Enter/«➤» → `fetch POST api/msg`), `<input type=file accept=image/*>`
  за кнопкой «📎», обработчик `paste` на документе (первый item типа
  image → POST `api/img`). Отправленное дублируется в ленту страницы
  пометкой «→ …» (для картинок — «→ 🖼 имя/скрин»). Ошибка отправки —
  существующая строка `err`. Токен добавляется как в аудио-POST.

## Решённые развилки

- Сырые POST-тела, не multipart (tiny_http не парсит) и не base64 (+33%).
- Декод и даунскейл на приёме, в памяти RGBA максимум ~30 постов —
  без записи на диск.
- Две секции, не общая лента: транскрипция отдельно, присланное отдельно.
- Просмотр без действий (не копируем, не сохраняем).
- Пуша со страницы в браузер нет — страница только шлёт (ws не нужен).

## Ошибки

- Тело больше лимита → `413`, картинка не декодится/чужой тип → `415`,
  пустой текст → `400`; страница показывает «не отправилось» в `err`.
- Окно не открыто — посты всё равно копятся в `Shared` (кап 30),
  увидишь при открытии.

## Тесты

- Юнит: маппинг `Content-Type` → допустимый формат; кап ленты
  (31-й пост вытесняет первый); геометрия даунскейла (вписывание
  4000×3000 в 1600×1600 → 1600×1200, мелкая картинка не растёт).
- E2E (существующий `#[ignore]`): POST текста и однопиксельного PNG
  с токеном → оба появляются в `Shared.posts`; `413`/`415` на
  негабарит/мусор.
