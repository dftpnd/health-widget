# Автопилот: цепочка аккаунтов по завершении откликов

## Задача

Когда бот в фазе `--apply` заканчивает отклики для одного аккаунта (процесс
`autopilot run --apply --profile X` завершается сам, исчерпав релевантный пул
вакансий), виджет должен переключиться на следующий аккаунт и продолжить
отклики, а не останавливаться. Останов — только когда полный круг по всем
профилям не дал ни одного нового отклика (общий пул исчерпан) либо все
профили упёрлись в дневной лимит.

## Что уже есть

- `maybe_rotate_profile` (`src/main.rs:745`, вызывается каждый тик из
  `update`): пока `--apply` **работает**, крутит профиль по кругу при
  достижении дневного лимита или после батча в `APPLY_BATCH_SIZE = 42`
  откликов. Остаётся без изменений.
- Обработчик выхода процесса (`src/main.rs:2620`): при `!alive()` для
  `Phase::Scan`/`ScanAll` цепляет `Phase::Enrich`; для `Phase::Apply` сейчас
  делает `want = None` и ставит статус «автопилот остановлен». **Это и есть
  точка правки.**

## Пробел

Когда процесс `--apply` завершается сам (пул для аккаунта исчерпан раньше, чем
набрался батч 42 или дневной лимит), `maybe_rotate_profile` не срабатывает
(ни `over_limit`, ни `batch_done`), а обработчик выхода просто останавливает
автопилот. Переход к следующему аккаунту не происходит.

## Дизайн

### Данные

Новое поле в `AutopilotState` (`src/main.rs:88`):

- `apply_idle: HashSet<String>` — профили, чей последний выход по `--apply`
  дал ноль новых откликов. Очищается при любом прогрессе. Служит детектором
  «круга без новых откликов».

Инициализация — пустое множество (в конструкторе состояния рядом с
`batch_baseline: 0`).

### Общий хелпер round-robin

Вынести из `maybe_rotate_profile` инлайн-подбор следующего профиля
(текущие строки 759–772) в метод:

```
fn next_eligible_profile(&self, from: &str) -> Option<String>
```

Обходит `PILOT_PROFILES` по кругу от `from` (индексы `1..=n`, последний шаг —
снова `from`, чтобы можно было продолжить тем же профилем новым батчем) и
возвращает первый профиль, у которого по `stats-<profile>.json`
`daily_limit <= 0 || applied_today < daily_limit`; иначе `None`.

`maybe_rotate_profile` начинает использовать этот хелпер — поведение сохраняется
байт-в-байт (может вернуть `from` при замкнутом круге).

### Поток при выходе `--apply`

В обработчике выхода (`src/main.rs:2620`), ветка для `finished == Some(Phase::Apply)`
**и** `want == Some(Phase::Apply)` (пользователь не выключал отклики вручную):

1. `made = свежие_stats.applied_today − batch_baseline`
   (stats текущего профиля уже перечитаны в `maybe_reload` в начале тика).
2. `made > 0` → `apply_idle.clear()`; иначе `apply_idle.insert(текущий_профиль)`.
3. `next = next_eligible_profile(текущий_профиль)`.
4. **Стоп**, если `next.is_none()` (все упёрлись в лимит) **или**
   `next` уже в `apply_idle` (круг замкнулся без новых откликов):
   `want = None`, статус «все аккаунты обработали пул откликов».
5. Иначе — переключение: `profile = next`, сброс `batch_baseline` из
   `stats-<next>.json` (как в существующей ротации), `stats = None`,
   `stats_mtime = None`, `scan_mtime = None`; `want` остаётся `Some(Apply)`;
   вызов `reconcile_pilot()` (стартует `--apply` для нового аккаунта).

Если `want != Some(Apply)` (пользователь выключил) — поведение как раньше:
останов со статусом «автопилот остановлен».

`apply_idle` также очищается при повторном включении «Откликов» из UI — там же,
где сейчас сбрасывается `batch_baseline` (переход `new_want → Some(Phase::Apply)`,
`src/main.rs:~2181` и `~2307`).

### Разделение решения и эффектов (для тестируемости)

Обработчик выхода сначала обновляет `apply_idle` (эффект), затем зовёт чистую
функцию решения:

```
enum ApplyChain { Switch(String), Stop }

fn decide_apply_chain(next: Option<&str>, apply_idle: &HashSet<String>) -> ApplyChain
```

Порядок в обработчике:

1. `made > 0` → `apply_idle.clear()`; иначе `apply_idle.insert(текущий_профиль)`.
2. `next = next_eligible_profile(текущий_профиль)`.
3. `decide_apply_chain(next, &apply_idle)`:
   - `next.is_none()` → `Stop`;
   - `next` уже в `apply_idle` (после вставки текущего) → `Stop`;
   - иначе → `Switch(next)`.

То есть `apply_idle` перед вызовом `decide` уже отражает исход текущего выхода.

## Телеметрия

Событие `pilot.apply_chain` при выходе `--apply`:

```json
{ "from": "<profile>", "made": <i64>, "next": "<profile|null>",
  "reason": "switch" | "idle_lap" | "all_limited" }
```

## Тесты

Юнит-тесты (в духе существующих в `src/pilot.rs`, без запуска процессов):

- `next_eligible_profile`: обход по кругу, пропуск залимиченных, возврат `from`
  при замкнутом круге, `None` когда все залимичены. (Потребует лёгкой
  тестируемой обёртки над подбором — по данным stats, а не по self.)
- `decide_apply_chain` (на вход — `apply_idle` уже после вставки текущего при
  `made == 0`):
  - `next == None` → `Stop`.
  - `next` не в `apply_idle` → `Switch(next)`.
  - `next` уже в `apply_idle` → `Stop`.
- сценарий обновления `apply_idle` + `decide` вместе:
  - `made > 0` → множество очищено → `Switch(next)`.
  - один профиль, пул пуст: `made == 0` → вставили `P` → `next == P` уже в
    множестве → `Stop` за один выход.
  - два профиля A(пусто)/B(есть работа): A выходит `made==0` → idle={A},
    `next=B` не в idle → `Switch(B)`; B выходит `made>0` → idle очищено →
    `Switch(A)`; когда оба пусты — второй пустой выход даёт `next` из idle →
    `Stop`.
