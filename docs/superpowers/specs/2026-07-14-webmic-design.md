# Веб-микрофон: страница на localhost → транскрипция в окно виджета

## Зачем

Со второго компа в локалке зайти браузером на страницу, говорить в его микрофон —
и видеть транскрипцию сразу в двух местах: на самой странице и в отдельном
плавающем окне виджета на основном компе.

## Как пользоваться

1. На основном компе — кнопка 🌐 в шапке виджета: тумблер сервера.
   При включении в буфер копируется `ssh -N -L 8787:localhost:8787 <user>@<ip>`
   и `http://localhost:8787`.
2. На втором компе: вставить ssh-команду в терминал, открыть URL в браузере.
   Микрофон работает без HTTPS, потому что для браузера это localhost.
3. Кнопка «Говорить» на странице → getUserMedia → звук идёт в виджет,
   текст — обратно на страницу и в окно `hw-web`.

## Архитектура

- **Сервер** — внутри виджета, `tiny_http` (единственная новая зависимость),
  слушает только `127.0.0.1:8787`, свой поток, `recv_timeout`-цикл.
  - `GET /*` — статика из `web/dist` (резолв от exe: `target/release/../../web/dist`;
    фолбэк `~/.local/share/health-widget/web`). Защита от path traversal.
  - `POST /api/audio?rate=<hz>&seq=<n>` — тело: сырой PCM f32 LE моно.
    Ответ: JSON `{ "finals": [..], "partial": ".." }` — свежие финальные реплики
    и текущая недоговорённая фраза.
- **Транскрипция** — третий `Transcriber` (whisper large-v3), канал «🌐 веб».
  Поднимается по требованию при первом аудио-чанке (rate берётся из запроса),
  гасится после 30 с без чанков (экономия ~4 ГБ VRAM). Финалы пишутся в общий
  журнал транскриптов (без привязки к колам, без WAV).
- **Окно** `hw-web` — плавающий always-on-top viewport по образцу `hw-clip`/`hw-chat`:
  лента финалов + partial серым + индикатор «клиент говорит/тишина».
  Позиция персистится (state: `web_x`/`web_y`), двигается и отслеживается через
  KWin-скрипт (маркер `HWWEB-GEOM`). Из захвата экрана исключено как и остальные
  окна виджета (resourceClass).
- **React-страница** — `web/` (vite + react, есть node v22). AudioWorklet собирает
  f32-чанки (реальный sampleRate контекста), главный поток шлёт их
  последовательными POST каждые ~250 мс и рисует ленту из ответов.
  Собирается `npm run build` в `web/dist`; правки фронта не требуют пересборки Rust.

## Решённые развилки

- Доступ к микрофону со второго компа: ssh-туннель (`-L`), не HTTPS.
- Транспорт: последовательные POST-чанки, не WebSocket (при 250 мс выигрыша нет).
- Канал — только окно + журнал; в колах не участвует (нет третьей WAV-дорожки).
- Сервер не рестартится автоматически при запуске виджета — только кнопкой.

## Ошибки

- Порт занят → кнопка не включается, статус-строка с ошибкой в шапке окна.
- Whisper не поднялся → в ответе `partial` = "⚠ распознавание недоступно".
- Клиент пропал (нет POST 30 с) → whisper гаснет, окно показывает «тишина».

## Тесты

- Юнит: разбор `rate` из query, сборка ssh-подсказки, декод PCM-тела,
  разбор `HWWEB-GEOM`, маппинг content-type статики.
- E2E: сервер поднят → curl статики; POST куска реального речевого WAV
  (calls/8/mic.wav) → в ответе появляются finals; телеметрия stt.start/stt.stop
  канала «🌐 веб».
